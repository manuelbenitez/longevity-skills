# Research Ingredient

Generate or update enriched ingredient JSON profiles. This skill is the only
producer of files in `<wiki_data_dir>/ingredients/`. It is driven by a per-book
dedup report and respects the four-bucket model:

| Bucket               | Action                                                |
|----------------------|-------------------------------------------------------|
| `new`                | Full enrichment + write a fresh profile               |
| `new-to-book`        | Load existing, append book's claims, preserve research, NO new LLM call |
| `existing-same-book` | Skip entirely                                         |
| `fuzzy`              | Skip — surfaced in dedup report for human review      |

## Implementation

The skill is implemented in `research-ingredient/enrich.py`. Run it directly:

```bash
.venv/bin/python research-ingredient/enrich.py --book <slug>
.venv/bin/python research-ingredient/enrich.py --book <slug> walnuts almonds  # manual override
.venv/bin/python research-ingredient/enrich.py                                # latest dedup report
```

## Configuration

- `ANTHROPIC_API_KEY` — required for live LLM calls (fresh enrichments only)
- `LONGEVITY_MODEL` — override model (default: `claude-haiku-4-5-20251001`)
- `LONGEVITY_WIKI_DATA_DIR` — override the target dir (otherwise read from
  `.longevity-skills.json[wiki_data_dir]`, default `./data`)

Per-skill model overrides in `.longevity-skills.json`:

```json
{ "models": { "research": "sonnet" } }
```

## Inputs

- `data/dedup/<slug>-dedup-report.json` — output of /dedup-ingredients. The
  enricher reads `book_slug`, `book_title`, and `author` directly from the
  report header — no extra args needed.
- `data/book-extracts/<slug>/ingredients-master.json` — for claim text per ingredient.
- `data/non-food-terms.txt` — shared filter; edit to add new exclusions.

## Output

`<wiki_data_dir>/ingredients/<slug>.json` per ingredient, conforming to
`schemas/ingredient.schema.json`:

```json
{
  "name": "Walnuts",
  "slug": "walnuts",
  "category": "nut",
  "aliases": [],
  "source_books": ["fontana", "longo"],
  "book_claims": [
    {
      "text": "Walnuts lower LDL cholesterol in adults at risk.",
      "mechanism": "ALA reduces hepatic VLDL output.",
      "recommendation": "1 ounce daily",
      "reference": "Estruch et al. 2018",
      "confidence": "high",
      "book_slug": "fontana"
    },
    {
      "text": "...",
      "book_slug": "longo"
    }
  ],
  "supplementary_research": [...],
  "flavor_profile": {...},
  "culinary_pairings": [...],
  "nutrient_highlights": [...],
  "synergies": [...],
  "last_updated": "2026-05-15T12:00:00Z",
  "last_researched": "2026-05-15T12:00:00Z",
  "research_status": "complete"
}
```

Every claim carries `book_slug`. The top-level `source_books[]` is recomputed on
every write from the claims — never edit it directly.

## Merge semantics (Decision ARCH-1)

When an existing profile is matched by a `new-to-book` bucket entry:

1. Load the existing profile.
2. Append the new claims, each tagged with the current `book_slug`.
3. Preserve `supplementary_research`, `flavor_profile`, `culinary_pairings`,
   `nutrient_highlights`, `synergies` byte-identical (no re-enrichment).
4. Recompute `source_books[]` from the merged claims.
5. Bump `last_updated`. Do NOT bump `last_researched` (no new research happened).
6. Validate against the schema. Write.

This is the fast path — no LLM call, no token cost.

## Validation and the .invalid/ tray

Every profile is validated against `schemas/ingredient.schema.json` before write.
If validation fails, the offending profile is written to `data/.invalid/<slug>-<timestamp>.json`
along with the validation diff. The skill prints the path and exits non-zero.

```bash
ls data/.invalid/   # forensics tray
```

The `data/.invalid/` directory is gitignored. Clean it manually when you're done
diagnosing.

## Failure modes

- **No dedup report:** Run /dedup-ingredients for the book first.
- **Missing manifest header in dedup report:** The report was generated by an
  old version of /dedup-ingredients. Re-run /dedup-ingredients.
- **API timeout / 5xx:** Auto-retried once with a 5s backoff. Second failure → exit.
- **Malformed JSON from LLM:** Whole batch goes to `.invalid/`. The next /research-ingredient
  run will re-attempt those ingredients.
- **Per-ingredient validation failure within a successful batch:** The one bad
  ingredient goes to `.invalid/`; the other 19 still write. Exit non-zero so the
  human notices.

## Token budget

| Batch size | Output tokens | Total per batch |
|------------|---------------|-----------------|
| 20 ingredients | ~600/ingredient | ~12k tokens (fits in max_tokens=16384) |

`max_tokens` is set to 16384. Earlier versions used 4096, which silently
truncated batches around ingredient 7 and wrote downstream entries with empty
research. Don't lower this without re-checking realistic output sizes.
